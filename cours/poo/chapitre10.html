🔹 Bonnes pratiques en POO

Pour écrire du code clair et maintenable, suis les principes SOLID :

  • S : Single Responsibility (une classe = une responsabilité).
  • O : Open/Closed (ouvert à l’extension, fermé à la modification).
  • L : Liskov Substitution (les sous-classes doivent être substituables).
  • I : Interface Segregation (interfaces spécifiques, pas générales).
  • D : Dependency Inversion (dépendre des abstractions).

🔹 Qu’est-ce qu’un design pattern ?

Un design pattern est une solution standard à un problème courant. Exemple : le Singleton garantit qu’une classe n’a qu’une seule instance.

💻 Exemple en Python

Voici une implémentation du pattern Singleton :


class Configuration:
    _instance = None
    
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance
    
    def __init__(self):
        self.param = "Config par défaut"

config1 = Configuration()
config2 = Configuration()
print(config1 == config2)  # Affiche : True (même instance)
print(config1.param)       # Affiche : Config par défaut

📌 Points clés

  • Les principes SOLID rendent le code robuste et maintenable.
  • Les design patterns comme Singleton résolvent des problèmes courants.
  • Écris des noms clairs et commente ton code pour plus de lisibilité.
